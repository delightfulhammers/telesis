# Telesis — Architecture
*By Delightful Hammers*
*Last updated: 2026-03-08*

---

## System Overview

Telesis is a Go CLI that manages structured project documentation and generates a `CLAUDE.md` context injection file. It produces a single static binary distributed as `telesis`.

The architecture follows a monorepo-ready layout to accommodate future agent binaries (e.g., Bop) without restructuring.

---

## Package Structure

```
telesis/
  cmd/
    telesis/
      main.go              ← entry point; calls cli.Execute()
  internal/
    cli/                   ← Cobra command definitions
      root.go
      init.go
      context.go
      adr.go
      tdd.go
      status.go
    config/                ← .telesis/config.yml read/write
    context/               ← CLAUDE.md generation from doc tree
    scaffold/              ← project initialization and file generation
    adr/                   ← ADR file management
    tdd/                   ← TDD file management
    status/                ← project status aggregation
  templates/               ← embedded document templates (go:embed)
  docs/
    VISION.md
    PRD.md
    ARCHITECTURE.md
    MILESTONES.md
    adr/                   ← ADR-NNN-slug.md
    tdd/                   ← TDD-NNN-slug.md
    context/               ← additional CLAUDE.md sections (freeform .md files)
  .telesis/
    config.yml
  CLAUDE.md                ← generated by `telesis context`
  go.mod
  go.sum
```

---

## Dependencies

Minimal by design:

- **`cobra`** — CLI framework (command parsing, help text, subcommands)
- **`viper`** — config file management (YAML read/write for `.telesis/config.yml`)
- **`go:embed`** — template embedding (zero runtime file dependencies)

No other external dependencies unless strictly necessary.

---

## Package Discipline

- **`cmd/telesis/`** contains only the entry point. A single `main.go` that calls `cli.Execute()`.
- **`internal/cli/`** contains Cobra command wiring — flag parsing, calling into business logic packages, printing output. This is the only package that imports Cobra.
- **`internal/{config,context,scaffold,adr,tdd,status}`** contain business logic. They know nothing about the CLI framework.
- **Nothing in `internal/` imports from `cmd/`.**
- **`templates/`** is embedded at compile time via `go:embed`. No runtime file I/O for templates.

## Data Flow

### `telesis init`

```
User input (flags)
  → internal/cli/init.go (parse flags, build config)
    → internal/scaffold.Scaffold(rootDir, config)
      → internal/config.Save(rootDir, config)
      → render templates → write doc files
      → internal/context.Generate(rootDir)
        → write CLAUDE.md
```

### `telesis context`

```
internal/cli/context.go
  → internal/context.Generate(rootDir)
    → internal/config.Load(rootDir)
    → scan docs/adr/ for ADR summaries
    → count docs/tdd/ TDD files
    → extract milestone from docs/MILESTONES.md
    → extract principles from docs/VISION.md
    → extract description from docs/VISION.md
    → extract conventions from docs/ARCHITECTURE.md
    → scan docs/context/*.md for additional sections
    → render claude.md.tmpl
    → write CLAUDE.md
```

### `telesis adr new <slug>`

```
internal/cli/adr.go (parse slug)
  → internal/adr.Create(rootDir, slug)
    → internal/adr.NextNumber(adrDir)
      → scan docs/adr/ for highest existing number
    → render adr.md.tmpl with number + slug
    → write docs/adr/ADR-NNN-slug.md
```

### `telesis tdd new <slug>` — same pattern as ADR.

### `telesis status`

```
internal/cli/status.go
  → internal/status.GetStatus(rootDir)
    → internal/config.Load(rootDir)
    → count ADRs, TDDs
    → read milestone info
    → check CLAUDE.md timestamp
  → format and print
```

## Error Handling

- Business logic packages (`internal/`) return errors. They never call `log.Fatal` or `os.Exit`.
- CLI commands (`internal/cli/`) catch errors and handle exit behavior.
- User-facing error messages are actionable: *"run `telesis init` first"*, not *"config not found"*.

## Template System

All generated files use Go templates embedded via `go:embed` in the `templates/` package.

Templates:
- `vision.md.tmpl` — VISION.md skeleton
- `prd.md.tmpl` — PRD.md skeleton
- `architecture.md.tmpl` — ARCHITECTURE.md skeleton
- `milestones.md.tmpl` — MILESTONES.md skeleton
- `adr.md.tmpl` — individual ADR
- `tdd.md.tmpl` — individual TDD
- `config.yml.tmpl` — .telesis/config.yml
- `claude.md.tmpl` — generated CLAUDE.md

Templates receive a data struct with project config and document metadata. The `telesis context` command is idempotent — same inputs always produce the same output.

### Template Completeness (CLAUDE.md)

The `claude.md.tmpl` template generates:
- **Project metadata** from `.telesis/config.yml`
- **About This Project** — extracted from VISION.md "The Vision" section
- **Quick Start** — static navigation links
- **Active Milestone** — extracted from MILESTONES.md (first `## MVP` section)
- **Recent Decisions** — scanned from `docs/adr/` (up to 5 most recent)
- **Key Documents** — static links with ADR/TDD counts
- **Principles** — extracted from VISION.md "Design Principles" section
- **Working Conventions** — extracted from ARCHITECTURE.md (Package Discipline, Error Handling, Testing Strategy sections)
- **Additional context** — all `.md` files from `docs/context/`, included verbatim in alphabetical order

The `docs/context/` directory is the escape hatch for project-specific sections that don't fit into the standard extraction points. Any markdown file placed there becomes a section in the generated CLAUDE.md.

## Key Design Decisions

Formal ADRs will track decisions going forward. Notable early decisions:

- **Monorepo-ready `cmd/telesis/main.go` layout** — supports adding future binaries (e.g., `cmd/bop/`) without restructuring. Bop migration deferred to post-MVP.
- **Cobra commands in `internal/cli/`** — keeps `cmd/telesis/main.go` trivial, isolates the Cobra dependency, enables clean ports-and-adapters separation.
- **`internal/scaffold/` instead of `internal/init/`** — avoids collision with Go's `init()` function convention.

---

## Testing Strategy

- **Unit tests** for all `internal/` packages. Table-driven where applicable.
- **Integration tests** for CLI commands against temp directories.
- **Don't test Cobra wiring** — test the underlying business logic functions.
- All tests operate on temp directories to avoid polluting the real filesystem.
